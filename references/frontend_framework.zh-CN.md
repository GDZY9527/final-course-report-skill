# 前端框架与消费级交付门槛

[English](frontend_framework.md) | [中文](frontend_framework.zh-CN.md)

## 结论

Flask 本身不是前端简陋的根本原因，但 Flask + Jinja + 少量 CSS 很容易被智能体降级成课程 demo。为避免再次生成简陋页面，除非用户明确指定其他技术栈，Python Web 项目默认使用 Django 构建完整前后端。

## 默认框架

非纯硬件课程、且项目使用 Python Web 技术时：

- 默认使用 Django。
- 必须包含 Django project、至少一个业务 app、templates、static、URL 路由、views、forms 或等价输入校验、错误处理页面。
- 若需要 API，使用 Django views 或 Django REST Framework；不得只用单文件脚本伪装完整后端。
- Flask 只能在用户明确指定、或项目明确是轻量 API 原型时使用；使用 Flask 仍必须满足消费级前端质量，不得以框架轻量为理由降低 UI。

## 消费级前端标准

前端不能只是“标题 + 卡片 + 表单 + 表格”。必须包含：

- 与项目领域绑定的首页工作台，而不是通用蓝色仪表盘。
- 清晰的信息架构：导航、主工作区、详情区、结果区、状态区。
- 至少 3 个真实业务页面或等价视图。
- 完整状态：加载、空数据、成功、失败、表单校验、后端错误。
- 真实数据驱动的图表、表格、详情面板和操作反馈。
- 响应式布局：桌面和移动端至少各检查一次。
- 可访问性：表单 label、按钮语义、焦点状态、颜色对比。
- 项目专属视觉：颜色、组件密度、数据展示方式与课程项目相匹配。

## 禁止项

- 单文件 Flask + Jinja + 粗糙 CSS 当作完整前后端。
- 所有课程都使用同一套蓝色 dashboard/card grid。
- 大面积空白、居中窄卡片、功能很少的 demo 页面。
- 前端出现未处理 JS 错误。
- 页面文字乱码。
- 页面截图未来自真实浏览器运行，却声称真实截图。

## 验证

交付前必须记录：

- 启动命令。
- 访问 URL。
- 页面清单。
- 真实浏览器截图路径。
- 浏览器控制台是否有错误。
- 移动端检查结果。

如果无法运行浏览器截图，必须如实标注为降级验证，不能声称“真实前端截图”。
