# 项目契约

[English](project_contract.md) | [中文](project_contract.zh-CN.md)

## 项目质量要求

项目应按可交付课程软件构建，而不是一次性演示脚本。

每个项目都必须提供：

- 清晰的目录结构
- README，包含安装、运行、测试、前端、结果、限制和扩展说明
- 使用 Python 时提供 `requirements.txt`
- 运行脚本或精确命令
- 按职责拆分的源码
- 测试或评价路径
- 相关日志和指标
- `results/` 中的可复现结果文件
- `results/report_notes.md`
- `results/design_summary.md`

## 前后端要求

除纯硬件课程外，新项目必须包含：

- 后端、服务层、算法层或核心程序
- 用于操作、可视化、监控、管理或结果展示的前端页面/应用
- 前端与项目真实结果之间的数据交换或稳定导出数据关联
- 来自真实页面或确定性渲染的前端截图

软硬结合课程应包含硬件输入/输出或模拟传感器/设备数据，并提供用于监控、配置、控制或结果可视化的前端。

## 结果契约

必须使用真实文件和真实运行结果：

- `results/metrics.json` 或等价测试/评价结果文件
- `results/run_log.txt` 或等价命令输出日志
- 项目目录结构图
- 系统/架构流程图
- 来自真实源码的关键代码截图
- 前端截图
- 评价图或测试汇总图

无法生成某项结果时，必须说明原因并提供真实兜底方案。

## 降级方案

当在线 API、大模型、编译器、浏览器自动化或外部服务不可用时：

- 优先使用较小的本地基线
- 在 README 和报告中保留原计划路线作为后续扩展
- 确保降级方案可运行、可复现
- 不得写成不可用组件已经成功运行

示例：

- BERT 不可用 -> TF-IDF 或传统机器学习基线，并补齐前端。
- LLM API 不可用 -> 离线检索/模板回答兜底，并补齐前端。
- 数据库服务不可用 -> SQLite 或文件存储兜底。
- 编译器不可用 -> 提供源码、构建说明和真实限制，并尽量验证周边脚本。
