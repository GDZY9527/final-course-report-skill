# 期末结课报告智能体指南

[English](AGENTS.md) | [中文](AGENTS.zh-CN.md)

当用户需要期末报告、结课报告、课程总结报告、配套实践项目、Word/PDF 报告、报告配图、项目截图或可复用报告生成流程时，使用本指南。

## 工作原则

- 先构建真实可运行、可复现的项目，再撰写项目实践章节。
- 除纯硬件课程外，所有新的纯软件或软硬结合项目都必须包含完整前端和后端，或等价的软件分层。
- 早期 NLP 示例没有前端只作为历史限制记录，后续新项目不得沿用该形态。
- 不得虚构实验结果、截图、代码片段、指标、构建成功、前端行为或 API 可用性。
- 当网络、API、模型下载、浏览器截图、编译器或依赖不可用时，优先提供可运行的离线降级方案，并在报告和交付说明中如实记录限制。
- 用户手动创建 conda 环境并安装依赖。智能体只负责编写 `requirements.txt` 和依赖说明，不默认静默安装包。
- 最小必需输入为课程名称、项目方向、Word 模板路径和 conda Python 路径。项目方向可以是用户明确指定的实践项目，也可以明确写“暂无明确项目目标，请智能体自行设计唯一性项目”。只有用户这样明确授权时，智能体才可自行选择项目。目标课程目录为可选项。姓名、学号、班级、教师为可选封面字段，未提供时留空，最后由用户在 Word 中自行填写。
- 优先做有根据的推断，少问问题。只有信息确实必要、无法从本地发现、且贸然假设风险较高时才询问用户。
- 中文 Markdown、JSON、CSV 和文本文件默认使用 UTF-8。

## 必须执行的流程

1. 检查目标目录、模板文档、现有项目文件和用户约束。
2. 创建或更新规范级项目，包括清晰模块、前端、后端/核心逻辑、数据、README、requirements、运行脚本、测试和 results。
3. 使用 `{conda_python}` 运行或记录精确验证命令。
4. 生成确定性的项目素材：目录树、架构/系统流程、代码截图、前端截图、指标图、日志和结果表。
5. 基于模板和真实项目输出撰写报告。
6. 默认加入参考文献或资料来源。
7. 有文档工具时，渲染并目检报告。
8. 打包前给出排除清单并等待确认。

## 最终输出要求

最终答复应列出：

- 报告路径
- 项目路径
- 使用的环境路径
- 生成的图表
- 验证命令和结果
- 已知限制
- 交付包内容或待确认的打包清单

## 本轮测试暴露的硬性改进

- 报告不得过短：生成前规划密度，生成后按 `references/report_density.md` 检查。
- 项目不得套模板：先写课程专属项目简报，再编码。
- 配图必须浅色、复杂、可打印，并嵌入 DOCX。
- 模板旧正文必须清理，旧课程关键词不得残留。
- 附录按用途组织，不得把代码、截图、日志混乱堆叠。
- 长任务维护 `results/execution_log.md`，记录命令输出和失败降级。
- 交付前运行可用审计脚本；失败项必须修复或如实列入限制。

## 2.0 回归测试后的新增硬规则

- 每份报告必须使用新项目；不得复用示例项目、历史测试项目或同课旧项目。用户提供项目方向时，应保留该方向并做唯一化细化，不得擅自替换为默认项目。
- 编码前必须生成 `results/project_brief.md`，并记录三个候选项目和避免复用的理由。
- 前端必须先生成 `results/frontend_design_brief.md`，最终页面要体现项目独特流程，不能所有项目都是同一套蓝色仪表盘。
- 用户未提供姓名、学号、班级、指导老师时，不在开头追问；封面字段留给用户最后在 Word 中填写。
- Word 报告必须严格遵守 `references/chengdu_neusoft_report_standard.zh-CN.md` 中的字体、字号、颜色、页边距、页眉页脚、标题、图题表题、公式和代码格式；模板样式与该规范冲突时，以该规范为准。
- 默认只创建“附录”标题，附录正文留空；正文密度不得靠附录灌水。
